# SQL Schema Design

*Database design decisions and relationships for question-level marking*

**For complete DDL:** See `data/db/schema.sql`

---

## Design Principles

### Subject-Agnostic Structure

All tables work identically for maths, English, chemistry, etc. Subject differences live in **content** (mark_type codes, descriptions), not **structure**.

**Adding new subjects:** No schema changes required. Add mark_types to reference table and subject-specific prompts to config.

### Question-Level Foundation

Questions are the fundamental unit of analysis and marking. Papers are collections of questions, not monolithic entities.

### Hierarchical Content Model

Questions and mark criteria use identical content block patterns:
- **Ordered blocks:** `display_order` for sequential content
- **Mixed content:** Text blocks and diagram blocks can interleave
- **Diagram constraint:** No two consecutive diagrams (combine into single image)

### Separation of Concerns

**Submission pipeline** (storage):
- `question_submissions` → `submission_images`
- Created when student uploads work
- Independent of marking

**Marking pipeline** (computation):
- `marking_attempts` → `question_marking_results`
- Can retry, use different models, run in batches
- Submissions persist even if marking fails

## Schema Overview

**Reference tables defined in:** `data/db/schema.sql`

Complete DDL with field types, constraints, and indexes available in schema file. Context docs focus on design decisions, not field listings.

### Reference Tables

**exam_types** - Exam specifications (templates)
- Key design: `paper_code` + `display_name` part of template, `total_marks` per instance
- Uniqueness: (exam_board, exam_level, subject, paper_code)
- Why: Same exam type sat multiple times per year (different dates = different instances)

**mark_types** - Marking criterion types (M, A, B, C, SPaG, AO3, etc.)
- Key design: Duplicated per exam type (not shared across subjects)
- Why: Duplication over complexity - handles edge cases without schema changes
- Scoped to: Each distinct paper specification defines its own mark codes

**llm_models** - Model normalization
- Why: 84% storage reduction on system's largest table
- Pattern: Normalize model identifiers to avoid string duplication

**students** - Temporary email auth
- Current: Email-based authentication (MVP)
- Future: Migrate to separate users table

### Core Tables - Question Structure

**papers** - Exam instances (dated sittings)
- Relationship: FK to exam_types (template → instance pattern)
- Uniqueness: (exam_type_id, exam_date) - prevents duplicate sittings

**questions** - High-level metadata
- Content location: `question_parts` and `question_content_blocks`
- Ordering: `question_number` determines display sequence

**question_parts** - Hierarchical structure (NULL part, (a), (b), (a)(i), etc.)
- NULL part: `part_letter = NULL, display_order = 0` for general context
- Every question: Has NULL part (container for general content)
- Expected answer: Optional LaTeX-formatted correct answer (from mark scheme)

**question_content_blocks** - Ordered text/diagram blocks
- Block types: 'text' or 'diagram' (XOR constraint)
- Diagram fields: `diagram_description` (for LLMs)
- Diagram paths: Derived from convention at API layer (not stored in database)
- Layout rule: No consecutive diagrams (enforced by trigger)

### Core Tables - Mark Scheme Structure

**mark_criteria** - High-level metadata
- Content location: `mark_criteria_content_blocks` (mirrors question structure)
- Critical: `criterion_index` must be contiguous 0, 1, 2, 3... (no gaps)
- Why contiguous: Enables array-index matching during LLM response validation
- Part reference: Every criterion references specific part via `question_part_id`
- NULL part reference: General guidance references NULL part
- GENERAL mark type: `marks_available = 0`, application guidance only

**mark_criteria_content_blocks** - Ordered text/diagram blocks
- Design: Mirrors `question_content_blocks` exactly
- Why: Science mark schemes show multiple diagrams with interspersed text
- Use cases: Apparatus setups, labeled diagrams, graph examples

**Dependencies:** `depends_on_criterion_index` for simple 1:1 dependencies
- Example: A1 accuracy mark depends on M1 method mark
- M1 scope: Simple dependencies only (complex follow-through deferred)
- Validation: Must reference earlier criterion (prevents circular deps)

### Core Tables - Submission & Marking

**question_submissions** - Student work (before marking)
- UUID: Generated by caller (enables idempotent operations)
- Design: Separate from marking (enables flexible timing)

**submission_images** - Multi-page student work
- Sequence: 1-based ordering (1, 2, 3...)
- Storage: R2 cloud paths (production) or local paths (evaluation)
- Cascade: Auto-delete when parent submission deleted

**marking_attempts** - All LLM interactions (success & failure)
- Status codes: success, parse_error, rate_limit, timeout, llm_error
- Token tracking: Separate input/output (output costs 4-5x input)
- Prompts: Stored for observability and debugging
- Retry support: Multiple attempts per submission

**question_marking_results** - Parsed LLM responses (success only)
- Normalization: Fully normalized (student_id via JOINs)
- One row per criterion per successful attempt

### M4 Tables - Paper Marking Context

**paper_attempts** - Full paper submissions
- Status: Derived from timestamps (in_progress → marking → complete)
- Lifecycle: Create → submit questions → submit paper → mark → grade → complete
- Retry: `inherited_from_attempt` copies latest submissions
- Soft deletion: `deleted_at` marks source after retry

**question_attempts** - Links submissions to papers
- Purpose: Track which questions submitted for each paper attempt
- Latest: MAX(created_at) determines current submission for question

**practice_question_attempts** - Individual question context
- Difference: No paper container, no grading, immediate feedback
- Cross-context: Each submission belongs to paper OR practice (never both)

**paper_results** - Final grades
- Calculation: Total marks → percentage → grade (via boundaries)
- Timing: Atomic with `completed_at` in same transaction
- Uniqueness: One result per paper attempt

**grade_boundaries** - Grade thresholds
- Loading: From paper JSON alongside structure
- Auto-add U: System appends U grade at 0 marks
- Lookup: Find highest grade where marks >= min_raw_marks

## Key Design Decisions

### Template vs Instance Pattern (exam_types → papers)

**Why separate:**
- Template (exam_type): Paper specification (code, display name, mark types)
- Instance (paper): Specific sitting (date, total marks, grade boundaries)
- Enables: Same template reused across multiple sittings

**Example:**
- Template: "1MA1/1H, Paper 1 (Non-Calculator)"
- Instance 1: Nov 2023, 80 marks
- Instance 2: May 2024, 80 marks

### Content Block Architecture

**Why interleaved text/diagrams:**
- Questions: "Given the diagram [diagram], calculate the area..."
- Mark schemes: "Accept setup A [diagram] or B [diagram] but not C [diagram]"

**Why no consecutive diagrams:**
- Forces intentional combination of related diagrams
- Prevents accidental diagram duplication
- Text blocks provide natural separators

### Separate Submission & Marking Pipelines

**Why separate:**
1. Different lifecycles: Storage immediate, marking flexible (batch, retry, overnight)
2. Cost protection: Submissions persist even if marking fails
3. Retry-friendly: Can re-mark without recreating submission
4. Flexibility: Mark immediately, batch later, or use different models

**Transaction boundaries:**
- Submission: One transaction per submission creation
- Marking: One transaction per marking attempt (special case: commit on MarkingError for observability)

### NULL Part Design

**Why every question has NULL part:**
- Container for general question context (not tied to specific part)
- Target for question-level marking guidance (e.g., "Accept equivalent forms")
- Simplifies queries: All parts exist as rows (no special-case logic)

### Paper Marking Contexts (M4)

**Why two contexts (paper vs practice):**
- Paper: Full workflow with grading, batch marking, immutable after completion
- Practice: Individual questions, immediate feedback, student can retry/delete

**Why separate tables:**
- Different lifecycles and constraints
- Clear separation of concerns
- Prevents mixing contexts (validation ensures submission belongs to one)

## Indexing Strategy

**Complete index definitions:** See `data/db/schema.sql`

**Index patterns:**
- **Composite indexes:** Common query patterns (student + question + date)
- **FK indexes:** JOIN performance (every foreign key indexed)
- **Temporal indexes:** `submitted_at DESC`, `attempted_at DESC` for latest-first queries
- **Partial indexes:** Failed attempts only (efficient failure analysis)

## Migration & Evolution

### Adding New Subjects

**No schema changes required:** Subject-agnostic structure requires only reference data (mark_types) and config changes (prompts, mappings).

### Future Expansions

**Deferred to post-M6:**
- Complex mark dependencies (alternative paths, multi-criterion follow-through)
- Working analysis storage (misconception tracking)
- Multi-user collaboration (teachers, cohort analysis)
- Advanced retry strategies (partial re-marking)
